## 1. Общее

**Base URL:** зависит от окружения, например `https://api.yourdomain.com`

**Формат данных:** все запросы и ответы — `application/json`.

**Content-Type для запросов с телом:**
```
Content-Type: application/json
```

**Дата/время:** все поля типа datetime передаются в формате ISO 8601:
```
2024-01-15T12:30:00
2024-01-15T12:30:00.000Z
```

## 2. Авторизация и токены

Большинство эндпоинтов требуют авторизации. Система использует два токена:

| Токен | Что это | Срок жизни | Где хранить |
|---|---|---|---|
| `access_token` | JWT-токен, подтверждает личность | 15 минут | В памяти (не в Keychain) |
| `refresh_token` | Случайная строка, обменивается на новую пару | 7 дней | В Keychain |

### Как передавать access_token

Добавляй заголовок `Authorization` к каждому запросу, требующему авторизации:

```
Authorization: Bearer <access_token>
```

### Логика обновления токенов

1. Делаешь запрос с `access_token`
2. Получаешь `401 Unauthorized` — токен истёк
3. Вызываешь `POST /auth/refresh` с `refresh_token`
4. Получаешь новую пару `access_token` + `refresh_token`
5. Сохраняешь оба, повторяешь исходный запрос

> После успешного рефреша **старый** `refresh_token` становится невалидным. Всегда сохраняй новый.

## 3. Аутентификация — `/auth`

### 3.1 Email: отправка кода

Отправляет 6-значный код подтверждения на email. Используется только при **регистрации**.

```
POST /auth/email/send-code
```

**Тело запроса:**
```json
{
  "email": "user@example.com"
}
```

**Успешный ответ `200`:**
```json
{
  "ok": true
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `400` | `"Too soon. Try again in N seconds"` | Повторная отправка раньше чем через 60 секунд |
| `400` | `"Email already registered"` | Такой email уже зарегистрирован |

---

### 3.2 Email: подтверждение регистрации

Завершает регистрацию. Проверяет код и создаёт аккаунт.

```
POST /auth/email/confirm-register
```

**Тело запроса:**
```json
{
  "email": "user@example.com",
  "password": "минимум8символов",
  "code": "123456"
}
```

| Поле | Тип | Ограничения |
|---|---|---|
| `email` | string | валидный email |
| `password` | string | от 8 до 256 символов |
| `code` | string | 6-значный код из письма |

**Успешный ответ `200` — `TokenPair`:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4...",
  "token_type": "bearer"
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `400` | `"Invalid or expired code"` | Неверный или устаревший код |
| `400` | `"Email already registered"` | Email уже занят |

---

### 3.3 Email: вход

Вход по email и паролю для уже зарегистрированного пользователя.

```
POST /auth/email/login
```

**Тело запроса:**
```json
{
  "email": "user@example.com",
  "password": "mypassword"
}
```

**Успешный ответ `200` — `TokenPair`:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4...",
  "token_type": "bearer"
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `401` | `"Invalid credentials"` | Неверный email или пароль |
| `401` | `"Account is inactive"` | Аккаунт деактивирован |

---

### 3.4 Google Sign In

Авторизация через Google. Получи `id_token` на клиенте через Google Sign In SDK и передай его на сервер.

```
POST /auth/google
```

**Тело запроса:**
```json
{
  "id_token": "<id_token из Google SDK>"
}
```

**Успешный ответ `200` — `TokenPair`:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4...",
  "token_type": "bearer"
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `401` | `"Invalid Google token"` | Токен не прошёл верификацию |
| `401` | `"Email already registered with another sign-in method"` | Email уже используется с другим методом входа |

> Если пользователь входит впервые — аккаунт создаётся автоматически.

---

### 3.5 Apple Sign In

Авторизация через Apple. Получи `identityToken` на клиенте через `ASAuthorizationAppleIDCredential` и передай его на сервер.

```
POST /auth/apple
```

**Тело запроса:**
```json
{
  "identity_token": "<identityToken из Apple SDK>"
}
```

**Успешный ответ `200` — `TokenPair`:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4...",
  "token_type": "bearer"
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `401` | `"Invalid Apple identity token"` | Токен не прошёл верификацию |
| `401` | `"Email already registered with another sign-in method"` | Email уже используется с другим методом входа |
| `401` | `"Apple account conflict"` | Тот же email привязан к другому Apple ID |

**Важно про Apple Sign In:**

Apple передаёт `email` только при **первой** авторизации пользователя. При последующих входах поле `email` будет пустым — это нормальное поведение Apple. Сервер корректно обрабатывает оба случая:
- Первый вход с email → создаётся аккаунт с email
- Повторный вход без email → поиск/создание по стабильному Apple `sub`

> Передавай `identityToken` как есть (строка в Base64). Поле называется `identity_token` (snake_case).

---

### 3.6 Обновление токенов

Обменивает `refresh_token` на новую пару токенов. Вызывай при получении `401` на любом защищённом эндпоинте.

```
POST /auth/refresh
```

**Тело запроса:**
```json
{
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4..."
}
```

**Успешный ответ `200` — `TokenPair`:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "новый_refresh_token_строка",
  "token_type": "bearer"
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `401` | `"Invalid refresh token"` | Токен не найден или истёк (7 дней) |

> После рефреша **сразу сохраняй** новый `refresh_token` — старый становится невалидным.

---

### 3.7 Выход

Инвалидирует `refresh_token`. После этого рефреш с данным токеном работать не будет.

```
POST /auth/logout
```

**Тело запроса:**
```json
{
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2ggdG9rZW4..."
}
```

**Успешный ответ `200`:**
```json
{
  "ok": true
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `401` | `"Invalid refresh token"` | Токен не найден |

---

### 3.8 Удаление аккаунта

Полностью удаляет аккаунт пользователя и **все** связанные данные: профиль, сессии, приёмы пищи, лайки, избранное.

```
DELETE /auth/account
```

**Заголовки:** требует авторизации.
```
Authorization: Bearer <access_token>
```

**Тело запроса:** не требуется.

**Успешный ответ `200`:**
```json
{
  "ok": true
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `401` | `"Not authenticated"` | Нет или неверный access_token |
| `401` | `"Invalid access token"` | Токен истёк или подпись неверна |

> После успешного удаления все токены пользователя становятся невалидными. Очисти локальное хранилище на клиенте.

## 4. Профиль — `/profile`

Профиль содержит медицинские параметры пользователя, необходимые для расчёта дозы инсулина.

### Поля профиля

| Поле | Тип | Описание |
|---|---|---|
| `target_glucose_mmol` | float | Целевой уровень глюкозы, ммоль/л |
| `insulin_sensitivity_factor` | float | Фактор чувствительности к инсулину (ISF) — на сколько ммоль/л снижает 1 ед. инсулина |
| `ic_ratio_breakfast` | float | Углеводный коэффициент на завтрак (г углеводов / 1 ед. инсулина) |
| `ic_ratio_lunch` | float | Углеводный коэффициент на обед |
| `ic_ratio_dinner` | float | Углеводный коэффициент на ужин |

При создании аккаунта профиль создаётся автоматически со всеми значениями `0`.

---

### 4.1 Получить профиль

```
GET /profile
```

**Заголовки:** требует авторизации.

**Успешный ответ `200`:**
```json
{
  "target_glucose_mmol": 5.5,
  "insulin_sensitivity_factor": 2.8,
  "ic_ratio_breakfast": 10.0,
  "ic_ratio_lunch": 12.0,
  "ic_ratio_dinner": 11.0
}
```

---

### 4.2 Обновить профиль

Полностью заменяет все поля профиля. Нужно передавать **все** поля.

```
PUT /profile
```

**Заголовки:** требует авторизации.

**Тело запроса:**
```json
{
  "target_glucose_mmol": 5.5,
  "insulin_sensitivity_factor": 2.8,
  "ic_ratio_breakfast": 10.0,
  "ic_ratio_lunch": 12.0,
  "ic_ratio_dinner": 11.0
}
```

| Поле | Тип | Ограничения |
|---|---|---|
| `target_glucose_mmol` | float | `>= 0` |
| `insulin_sensitivity_factor` | float | `>= 0` |
| `ic_ratio_breakfast` | float | `>= 0` |
| `ic_ratio_lunch` | float | `>= 0` |
| `ic_ratio_dinner` | float | `>= 0` |

**Успешный ответ `200`:** те же поля, что и в GET.

## 5. Блюда — `/dishes`

Блюдо (`Dish`) — шаблон еды с набором ингредиентов и их пищевой ценностью. Блюда могут быть **приватными** (видны только автору) или **публичными** (видны всем).

### Объект `Ingredient` (внутри блюда)

```json
{
  "name": "Гречка варёная",
  "weight_g": 150.0,
  "carbs_per_100g": 21.3
}
```

### Объект `DishResponse`

```json
{
  "id": 42,
  "author_user_id": 7,
  "title": "Гречневая каша с молоком",
  "is_public": true,
  "ingredients": [
    {
      "name": "Гречка варёная",
      "weight_g": 150.0,
      "carbs_per_100g": 21.3
    },
    {
      "name": "Молоко 2.5%",
      "weight_g": 100.0,
      "carbs_per_100g": 4.7
    }
  ],
  "created_at": "2024-01-15T12:00:00",
  "likes_count": 12,
  "is_liked": false,
  "is_favorited": true
}
```

---

### 5.1 Создать блюдо

```
POST /dishes
```

**Заголовки:** требует авторизации.

**Тело запроса:**
```json
{
  "title": "Гречневая каша с молоком",
  "is_public": true,
  "ingredients": [
    {
      "name": "Гречка варёная",
      "weight_g": 150.0,
      "carbs_per_100g": 21.3
    },
    {
      "name": "Молоко 2.5%",
      "weight_g": 100.0,
      "carbs_per_100g": 4.7
    }
  ]
}
```

| Поле | Тип | Описание |
|---|---|---|
| `title` | string | Название блюда |
| `is_public` | bool | `false` по умолчанию |
| `ingredients` | array | Список ингредиентов (не пустой) |
| `ingredients[].name` | string | Название ингредиента |
| `ingredients[].weight_g` | float | Вес в граммах (`> 0`) |
| `ingredients[].carbs_per_100g` | float | Углеводы на 100 г (`>= 0`) |

**Успешный ответ `200` — `DishResponse`**

## 6. Приёмы пищи — `/meals`

Приём пищи (`Meal`) — запись о том, что пользователь съел и когда. Содержит один или несколько **элементов** (`MealItem`), каждый из которых описывает отдельную позицию еды.

### Типы элементов приёма пищи

При создании meal каждый элемент в массиве `items` должен иметь поле `type`:

| Тип | Описание |
|---|---|
| `"manual_total"` | Ручной ввод — описание и суммарные углеводы |
| `"ingredients"` | Список ингредиентов с весом и углеводами каждого |
| `"dish"` | Готовое блюдо из базы, с указанием съеденного веса |

---

### 6.1 Создать приём пищи

```
POST /meals
```

**Заголовки:** требует авторизации.

**Тело запроса:**
```json
{
  "eaten_at": "2024-01-15T12:30:00",
  "items": [
    {
      "type": "manual_total",
      "description": "Пирожок с капустой",
      "total_carbs": 35.0
    },
    {
      "type": "ingredients",
      "ingredients": [
        {
          "name": "Рис варёный",
          "weight_g": 180.0,
          "carbs_per_100g": 28.0,
          "carbs_total": 50.4
        },
        {
          "name": "Куриная грудка",
          "weight_g": 120.0,
          "carbs_per_100g": 0.0,
          "carbs_total": 0.0
        }
      ]
    },
    {
      "type": "dish",
      "dish_id": 42,
      "eaten_weight_g": 250.0
    }
  ]
}
```

**Описание типов элементов:**

**`manual_total`** — ручной ввод суммарных углеводов:
```json
{
  "type": "manual_total",
  "description": "Пирожок с капустой",
  "total_carbs": 35.0
}
```

**`ingredients`** — список ингредиентов, углеводы суммируются:
```json
{
  "type": "ingredients",
  "ingredients": [
    {
      "name": "Рис варёный",
      "weight_g": 180.0,
      "carbs_per_100g": 28.0,
      "carbs_total": 50.4
    }
  ]
}
```

> `carbs_total` для каждого ингредиента нужно посчитать на клиенте: `weight_g * carbs_per_100g / 100`

**`dish`** — блюдо из базы с указанием съеденного веса. Сервер сам пересчитает углеводы пропорционально весу:
```json
{
  "type": "dish",
  "dish_id": 42,
  "eaten_weight_g": 250.0
}
```

**Успешный ответ `200` — `MealResponse`:**
```json
{
  "id": 15,
  "eaten_at": "2024-01-15T12:30:00",
  "total_carbs": 121.4
}
```

**Ошибки:**
| Статус | detail | Причина |
|---|---|---|
| `400` | `"Dish not found"` | Блюдо с таким `dish_id` не существует |
| `403` | `"Access denied to this dish"` | Блюдо приватное и не принадлежит пользователю |
| `400` | `"Invalid meal item type"` | Передан неизвестный тип элемента |

```json

  {
    "id": 31,
    "type": "manual_total",
    "items": [

      }

  },

    "type": "ingredients",
    "items": [
      {
        "name": "Рис варёный",

    "created_at": "2024-01-15T12:30:05"
```

**Структура поля `items` зависит от `type`:**

## 7. Коды ошибок

### Общие HTTP-статусы

| Статус | Значение |
|---|---|
| `200` | Успех |
| `400` | Неверный запрос (ошибка валидации или бизнес-логики) |
| `401` | Не авторизован (нет токена, токен истёк или неверен) |
| `403` | Нет доступа (авторизован, но нет прав) |
| `404` | Ресурс не найден |
| `422` | Ошибка валидации тела запроса (FastAPI автоматически) |

### Формат ошибки

```json
{
  "detail": "Текст ошибки"
}
```

Для `422` (validation error) формат отличается:
```json
{
  "detail": [
    {
      "type": "missing",
      "loc": ["body", "email"],
      "msg": "Field required",
      "input": {}
    }
  ]
}
```

### Алгоритм обработки `401`

```
1. Получили 401 на защищённом эндпоинте
2. Вызываем POST /auth/refresh с refresh_token
   - Если 200 → сохраняем новые токены, повторяем запрос
   - Если 401 → refresh_token тоже истёк → отправляем пользователя на экран входа
```
