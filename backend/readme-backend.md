# DiabetAI API — документация бекенда для фронтенда

Документ описывает **REST API** приложения DiabetAI (учёт глюкозы, инсулина, блюд и приёмов пищи). Его можно передать разработчику **мобильного (Swift)** или **веб**-клиента как единый источник правды по контрактам.

---

## 1. Технологии и принцип работы

| Компонент | Описание |
|-----------|----------|
| **Фреймворк** | FastAPI (Python 3.11+) |
| **База данных** | PostgreSQL (через SQLAlchemy 2, sync-сессии) |
| **Авторизация** | Два токена: **access** (JWT) + **refresh** (случайная строка, хранится в БД) |
| **Пароли** | bcrypt (пароль предварительно нормализуется через SHA-256, чтобы обойти лимит bcrypt 72 байта) |
| **Документация в рантайме** | Swagger UI: `GET /docs`, ReDoc: `GET /redoc` |

### 1.1. Общая схема авторизации

1. Пользователь проходит один из потоков входа (**email**, **Google**, **Apple**).
2. Сервер возвращает **`TokenPair`**: `access_token`, `refresh_token`, `token_type: "bearer"`.
3. Все защищённые запросы отправляются с заголовком:
   ```http
   Authorization: Bearer <access_token>
   ```
4. **Access-токен** — JWT с ограниченным сроком жизни (`ACCESS_TOKEN_EXPIRE_MINUTES`, по умолчанию 15 минут). В payload есть поля `sub` (id пользователя), `type: "access"`, `exp`.
5. Когда access истёк, клиент **не** должен снова логиниться: нужно вызвать **`POST /auth/refresh`** с телом `{ "refresh_token": "..." }`.
6. **Важно:** ответ `/auth/refresh` — снова полный **`TokenPair`**. Сервер **ротирует refresh**: старый refresh после успешного ответа **недействителен**. Клиент обязан **сохранить новый `refresh_token`** (Keychain / Secure Storage).
7. Выход: **`POST /auth/logout`** с `{ "refresh_token": "..." }` — сессия удаляется из БД.

### 1.2. Ошибки и формат ответа

- Успешные ответы — JSON с телом, описанным ниже по эндпоинтам.
- Ошибки FastAPI / `HTTPException`: обычно **`{"detail": ...}`**, где `detail` — строка или список объектов валидации (для 422).

Типичные коды:

| Код | Когда |
|-----|--------|
| **400** | Неверные данные, неверный код регистрации, ограничение повторной отправки кода email |
| **401** | Нет/битый access, неверный refresh, неверные учётные данные при логине |
| **403** | Пользователь деактивирован (`is_active`), доступ к чужому приватному ресурсу |
| **404** | Сущность не найдена |
| **422** | Ошибка валидации тела запроса (Pydantic) |

## 2. Переменные окружения (`.env`)

Все значения читаются из **`core/config.py`** (pydantic-settings). Поля **обязательны**, если у них нет значения по умолчанию.

### 2.1. Приложение и БД

| Переменная | Пример / назначение |
|------------|---------------------|
| `DATABASE_URL` | `postgresql+psycopg2://user:pass@host:5432/dbname` |
| `APP_TITLE` | Название в OpenAPI (по умолчанию `DiabetAI API`) |
| `ENV` | `dev` / `prod` (информативно) |
| `ROOT_PATH` | Если API за прокси с префиксом (например `/api`) |

### 2.2. JWT

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `JWT_SECRET_KEY` | — | Секрет для подписи access JWT (**в продакшене — длинная случайная строка**) |
| `JWT_ALGORITHM` | `HS256` | Алгоритм |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | `15` | Жизнь access-токена |
| `REFRESH_TOKEN_EXPIRE_DAYS` | `7` | Срок жизни refresh-сессии в БД |

### 2.3. Email (SMTP)

| Переменная | Описание |
|------------|----------|
| `SMTP_HOST`, `SMTP_PORT` | Сервер и порт (часто 465 для SSL) |
| `SMTP_USER`, `SMTP_PASS` | Учётные данные |
| `FROM_EMAIL` | Адрес отправителя писем с кодом |

### 2.4. Google Sign-In

| Переменная | Описание |
|------------|----------|
| `GOOGLE_IOS_CLIENT_ID` | iOS client ID (в коде проверки токена может не использоваться напрямую — см. ниже) |
| `GOOGLE_WEB_CLIENT_ID` | Web client ID |
| `GOOGLE_WEB_CLIENT_SECRET` | Секрет веб-клиента |
| `GOOGLE_ALLOWED_AUDS` | **Строка через запятую** допустимых `aud` из Google ID token (например iOS + Web client id). Именно по этому списку сверяется audience после верификации подписи токена |

### 2.5. Apple Sign-In

| Переменная | Описание |
|------------|----------|
| `APPLE_BUNDLE_ID` | **Обязательно** для проверки `identity_token`: должен совпадать с `aud` в JWT (Bundle ID iOS-приложения) |
| `APPLE_TEAM_ID`, `APPLE_KEY_ID`, `APPLE_PRIVATE_KEY_PATH` | Зарезервированы под серверные сценарии (например генерация client secret для web); **текущая реализация входа по `identity_token` использует в первую очередь `APPLE_BUNDLE_ID`** |

### 4.1. Auth — `/auth`

#### `POST /auth/email/send-code`

Регистрация по email: отправка 6-значного кода на почту.

| | |
|--|--|
| **Auth** | Нет |
| **Body** | `{ "email": "user@example.com" }` (`EmailStr`) |
| **200** | `{ "ok": true }` |
| **400** | Например: пользователь уже существует; нужно подождать перед повторной отправкой (**~60 секунд** с одного email) |

---

#### `POST /auth/email/confirm-register`

Подтверждение email и завершение регистрации.

| | |
|--|--|
| **Auth** | Нет |
| **Body** | `{ "email": "...", "password": "...", "code": "..." }` — пароль **8–256** символов |
| **200** | **`TokenPair`** (см. §4.1.1) |
| **400** | Неверный код, пользователь уже есть |

---

#### `POST /auth/email/login`

| | |
|--|--|
| **Auth** | Нет |
| **Body** | `{ "email": "...", "password": "..." }` |
| **200** | **`TokenPair`** |
| **401** | Неверные данные; аккаунт деактивирован |

---

#### `POST /auth/google`

| | |
|--|--|
| **Auth** | Нет |
| **Body** | `{ "id_token": "<Google ID JWT>" }` |
| **200** | **`TokenPair`** |
| **401** | Невалидный токен, неверный audience, нет email в токене, аккаунт неактивен |

Сервер верифицирует ID token Google и проверяет, что `aud` входит в `GOOGLE_ALLOWED_AUDS`.

---

#### `POST /auth/apple`

| | |
|--|--|
| **Auth** | Нет |
| **Body** | `{ "identity_token": "<Apple identity JWT>" }` |
| **200** | **`TokenPair`** |
| **401** | Невалидный токен, конфликт аккаунта, неизвестный Apple-аккаунт (первый вход без email в токене и без записи в БД) и т.д. |

Проверка: подпись Apple (JWKS), `iss`, `aud` = `APPLE_BUNDLE_ID`, пользователь связывается по **`sub`** и при необходимости по email.

**База данных:** в таблице `users` должно быть поле **`apple_sub`** (уникальный идентификатор Apple). Если деплой на существующую БД без миграции — нужно выполнить SQL миграцию (см. раздел 7).

---

#### `POST /auth/refresh`

Обновление пары токенов.

| | |
|--|--|
| **Auth** | Нет (передаётся refresh в теле) |
| **Body** | `{ "refresh_token": "..." }` |
| **200** | **`TokenPair`** — **новый** `access_token` и **новый** `refresh_token` |
| **401** | Сессия не найдена, срок refresh истёк, пользователь удалён/деактивирован |

Клиент **должен заменить** сохранённый refresh на новый из ответа.

---

#### `POST /auth/logout`

| | |
|--|--|
| **Auth** | Нет |
| **Body** | `{ "refresh_token": "..." }` |
| **200** | `{ "ok": true }` |
| **401** | Refresh не найден |

---

#### §4.1.1. Модель `TokenPair`

```json
{
  "access_token": "string",
  "refresh_token": "string",
  "token_type": "bearer"
}
```

#### `GET /profile`

|--|--|

  "ic_ratio_lunch": 0.0,

| | |
|--|--|
| **Body** | Те же поля, что в ответе, все числа **`>= 0`** |
| **200** | Актуальный `ProfileResponse` |

### 4.3. Блюда — `/dishes`

Все эндпоинты с **`Authorization: Bearer`**.

#### `POST /dishes`

Создание блюда.

**Body (`DishCreateRequest`):**

```json
{
  "title": "string",
  "is_public": false,
  "ingredients": [
    {
      "name": "string",
      "weight_g": 100.0,
      "carbs_per_100g": 12.5
    }
  ]
}
```

- `weight_g` > 0, `carbs_per_100g` ≥ 0.

**200:** `DishResponse` (см. ниже). Для только что созданного блюда счётчики лайков/избранного обычно **0 / false**.

---

#### `GET /dishes`

Список блюд: **публичные** (`is_public: true`) **или** принадлежащие текущему пользователю. Сортировка: по `created_at` убыв.

**200:** `DishResponse[]`

---

#### `GET /dishes/{dish_id}`

Одно блюдо. Приватное чужое — **403**.

**200:** `DishResponse`

---

#### `PUT /dishes/{dish_id}`

Обновление (только автор).

**Body (`DishUpdateRequest`):** как создание, но `is_public` обязателен явно.

**200:** `DishResponse`  
**403/404** — не ваше блюдо / не найдено

---

#### `DELETE /dishes/{dish_id}`

Только автор.

**200:** `{ "ok": true }`

---

#### `DishResponse` (общая модель)

```json
{
  "id": 1,
  "author_user_id": 1,
  "title": "string",
  "is_public": false,
  "ingredients": [],
  "created_at": "2026-04-06T12:00:00",
  "likes_count": 0,
  "is_liked": false,
  "is_favorited": false
}
```

`ingredients` в БД хранятся как JSON-массив объектов с полями, совместимыми с `Ingredient`.

---

#### Лайки

| Метод | Путь | Тело | Успех |
|-------|------|------|--------|
| POST | `/dishes/{dish_id}/like` | нет | `{ "ok": true }` (повторный лайк может «молча» не создаться из-за уникального ограничения) |
| DELETE | `/dishes/{dish_id}/like` | нет | `{ "ok": true }` |

---

#### Избранное

| Метод | Путь | Успех |
|-------|------|--------|
| POST | `/dishes/{dish_id}/favorite` | `{ "ok": true }` |
| DELETE | `/dishes/{dish_id}/favorite` | `{ "ok": true }` |

### 4.4. Приёмы пищи — `/meals`

Все с **`Authorization: Bearer`**.

#### `POST /meals`

Создание приёма с позициями разных типов.

**Body (`MealCreateRequest`):**

```json
{
  "eaten_at": "2026-04-06T12:30:00",
  "items": []
}
```

Элементы `items` — дискриминированные по полю **`type`**:

1. **`manual_total`** — ручной ввод углеводов:
   ```json
   { "type": "manual_total", "description": "...", "total_carbs": 45.0 }
   ```

2. **`ingredients`** — список ингредиентов со снимком:
   ```json
   {
     "type": "ingredients",
     "ingredients": [
       {
         "name": "x",
         "weight_g": 100,
         "carbs_per_100g": 10,
         "carbs_total": 10
       }
     ]
   }
   ```

3. **`dish`** — расчёт из сохранённого блюда:
   ```json
   { "type": "dish", "dish_id": 1, "eaten_weight_g": 200.0 }
   ```

Доступ к блюду: только **публичное** или **своё**; иначе **403**. Некорректные данные блюда — **400**.

**200 (`MealResponse`):**

```json
{
  "id": 1,
  "eaten_at": "2026-04-06T12:30:00",
  "total_carbs": 123.45
}
```

---

#### `GET /meals`

Список приёмов **текущего пользователя**, по `eaten_at` убыв.

**200:** `MealResponse[]`

---

#### `GET /meals/{meal_id}/items`

Позиции приёма (только свой приём).

**200:** массив **`MealItemResponse`:**

```json
{
  "id": 1,
  "type": "ingredients",
  "items": [],
  "total_carbs": 10.0,
  "created_at": "2026-04-06T12:30:00"
}
```

**404** — чужой или несуществующий приём.

## 5. Рекомендации для клиента (Swift / мобильное приложение)

1. Хранить **`access_token`** в памяти; **`refresh_token`** — в Keychain.
2. Перед каждым запросом к защищённым API: если access истёк (по `exp` или по 401) — вызвать **`/auth/refresh`**, обновить оба токена, повторить запрос.
3. После **`/auth/refresh`** всегда сохранять **новый** `refresh_token`.
4. Даты в JSON — **ISO 8601**; убедитесь, что декодер корректно обрабатывает timezone.
5. Apple: передавайте **`identity_token`** с устройства после ASAuthorization.
6. Google: передавайте **`id_token`**, полученный от Google Sign-In.

## 6. Готовность к «боевому» деплою — честный чеклист

Ниже — что **уже есть** и что стоит сделать **до** полноценного продакшена.

### Уже заложено в коде

- Разделение access/refresh, ротация refresh, logout.
- Проверка Google/Apple токенов на сервере (не доверять только клиенту).
- Пароли хэшируются; JWT подписывается секретом.
- Привязка данных пользователя к `user_id`; доступ к чужим приватным блюдам в приёмах пищи ограничен.

### Обязательно перед продом (операционка)

| Задача | Зачем |
|--------|--------|
| **Надёжный `JWT_SECRET_KEY`** | Иначе подделка access-токенов |
| **HTTPS** у клиента до API | Защита токенов и данных |
| **Резервное копирование БД** | Потеря данных |
| **Мониторинг и логи** (хотя бы stdout + ротация, алерты по 5xx) | Диагностика |
| **Миграции схемы** (Alembic или аналог) | `create_all` **не обновляет** существующие таблицы; новые колонки (`apple_sub` и др.) нужно накатывать SQL/миграциями |

### Желательно улучшить (не блокирует запуск, но снижает риски)

| Тема | Комментарий |
|------|-------------|
| Коды email в памяти процесса | При нескольких инстансах или рестарте — сбой; для масштаба — Redis/БД |
| Глобальный rate limit / WAF | Защита от перебора кодов и DDoS |
| Единый часовой пояс UTC в БД | Сейчас часто используется `datetime.utcnow()` — согласовать с фронтом |
| Ограничение числа сессий на пользователя | Сейчас при каждом логине добавляется новая строка `sessions` |

### Итог

**Запустить API в продакшене можно**, если закрыты секреты, HTTPS, бэкапы, миграции БД и базовый мониторинг. Для **высокой нагрузки и многоролевого масштабирования** понадобятся доработки (хранилище кодов, миграции, rate limiting, политика сессий).

## 7. SQL для колонки Apple (если БД уже существовала до появления `apple_sub`)

```sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS apple_sub VARCHAR UNIQUE;
CREATE INDEX IF NOT EXISTS ix_users_apple_sub ON users (apple_sub);
```

Проверьте синтаксис под вашу СУБД (PostgreSQL пример выше).
